    Embedding

    July 7, 2026
    Rag, Llm

    Embedding#

    Embedding（嵌入）是把一段文本、图像或音频映射成「稠密向量」的过程，目标是让语义相近的内容在向量空间里也距离相近。大模型语境下最常说的就是文本 Embedding：由 Embedding 模型（如 Qwen3 Embedding、bge、text-embedding-3-small）把句子切成 768/1024/1536 维的浮点向量，供下游做相似度检索、聚类或分类。

    出现动机#

    大模型处理的是 token 序列，人类日常打交道的是文字、图片、音频。要让模型「理解」这些原始数据，需要先把它们转成连续的向量表示——这就是 Embedding 的由来。早期的 Embedding 是 Word2Vec / GloVe 这类静态词向量，如今主流是「句子级 / 段落级」的稠密向量，由 BERT、bge、Qwen3 Embedding、text-embedding-3-small 等大模型生成，维度从 256 到 3072 不等，核心目标都是「语义相近的输入在向量空间里距离也相近」。

    优点缺点#

    优点：①把语义层面的「相似 / 不相似」转化为可计算的「距离 / 角度」，让检索、聚类、推荐等下游任务有了统一表示；②同一个 Embedding 模型可以服务多种业务（语义检索、去重、聚类、分类），投入产出比高；③开源生态丰富（bge、Qwen3 Embedding、M3E 等），私有部署门槛低。缺点：①对长文本处理不友好，超出上下文窗口的文本需要先切分；②不同模型产出的向量空间不通用，跨模型比对意义有限；③对中文古文、代码、专业公式等领域的 Embedding 质量参差，需要按业务场景选型。

    Milvus

    July 7, 2026
    Rag, Infra

    Milvus#

    Milvus 是一款面向大规模向量检索场景设计的开源向量数据库，由 Zilliz 开源，主打高吞吐、低延迟的 ANN 检索。它支持 IVF/HNSW/DiskANN 等多种索引，提供标量字段与向量字段混合检索、多副本容灾、GPU 加速等特性，是当前 RAG / 推荐系统 / 以图搜图等场景里最常用的向量库之一。

    出现动机#

    在 RAG 与推荐系统等场景里，向量规模经常达到百万到十亿级别，传统数据库的「精确 KNN」会随着向量维度上升出现维度灾难，召回延迟从毫秒级掉到秒级甚至分钟级。Milvus 由 Zilliz 开源，专为大规模 ANN 检索而设计：内置 IVF / HNSW / DiskANN / GPU 索引，支持标量字段过滤 + 向量字段相似度召回的混合查询，提供多副本容灾、分片水平扩展、Collection / Partition / Alias 等数据组织能力。

    优点缺点#

    优点：①亿级向量毫秒级召回，是工业级 RAG 检索阶段的常见选择；②支持 CPU / GPU / 混合部署，可按成本灵活切换；③生态完善，与 LangChain、LlamaIndex、Dify 等主流框架都有官方集成。缺点：①独立部署一套 Milvus 集群的运维成本远高于「PostgreSQL 加 pgvector 扩展」；②对超大规模向量（百亿级）需要分片、归档、热点调优等额外工程；③相比关系数据库，其事务能力很弱，不适合作为唯一存储。

    语义检索

    July 7, 2026
    Rag

    语义检索#

    语义检索是基于「意思相近」而不是「字面匹配」的检索方式，核心思路是：把查询和文档都 Embedding 成向量，再用向量相似度（如余弦距离）排序。和传统 BM25 这种关键词检索相比，语义检索擅长处理同义改写、跨语言查询、抽象表达等场景；实际 RAG 系统往往把语义检索 + 关键词检索混合使用，兼顾召回率和精确度。

    出现动机#

    关键词检索（BM25、TF-IDF）擅长精确匹配字面词，但对「同义改写」「跨语言」「抽象表达」无能为力：用户问「如何退订会员」，系统里只写「取消订阅流程」，关键词检索可能完全搜不到。语义检索（Semantic Retrieval）通过 Embedding 把查询和文档都映射到同一向量空间，用向量相似度代替字面匹配，从根本上解决这个问题。

    优点缺点#

    优点：①能召回同义、近义、跨语言的表达，对自然语言查询更友好；②对长尾查询、抽象问题效果远好于纯关键词检索；③和关键词检索结合（Hybrid Search）可以兼顾召回率与精确度。缺点：①对专有名词、型号、代码等字面敏感内容召回不稳定；②Embedding 模型本身的偏差会带到检索结果里；③纯向量检索的可解释性比 BM25 差，难以解释「为什么召回这条」。

    重排序

    July 7, 2026
    Rag, Llm

    重排序#

    重排序（Reranking / Rerank）是对初检召回的候选文档进行「二次精排」的过程，通常由一个 Cross-Encoder 模型完成：它会把「查询 + 候选文档」一起送进模型，输出更精细的相关性分数。Reranker 一般比 Embedding 用的双塔模型贵得多，因此 RAG 系统常用「Embedding 召回 Top-K → Reranker 精排 Top-N」的两阶段架构，在效果与成本之间取得平衡。

    出现动机#

    Embedding 检索用的是「双塔模型」：查询与文档分别编码成向量再算相似度，速度快但精度有限——它本质上是「查询侧与文档侧相互独立」的结构，没法捕捉两者深度交互。重排序（Reranking / Rerank）就是为了弥补这一环：把「查询 + 候选文档」一起送进一个 Cross-Encoder 模型，输出更精细的相关性分数，作为最终排名的依据。

    优点缺点#

    优点：①精度显著高于纯双塔召回，复杂问题、长 query 受益尤其明显；②一般只在 Top-K（20~100）候选上跑，开销可控；③常作为「两阶段索引」的第二阶段，与 Embedding 召回互补。缺点：①推理成本远高于 Embedding 检索（每对 query+doc 都要重新前向）；②对超长文档要先做切片后再分别打分，否则会超上下文窗口；③不同 Reranker 在中文 / 代码 / 跨语言等场景下表现差异大，需要选型评估。

    两阶段索引

    July 7, 2026
    Rag

    两阶段索引#

    两阶段索引是 RAG 检索阶段的经典架构：第一阶段用 Embedding 做「粗排召回」，从百万级语料中快速拉回 Top-K 候选；第二阶段用 Reranker 做「精排」，把候选数压到 Top-N 再交给 LLM。它的本质是用廉价的 ANN 检索换召回广度，用昂贵的 Cross-Encoder 换精度，是工业级 RAG 系统的事实标准。

    出现动机#

    在工业级 RAG 系统里，文档语料往往动辄百万级起步，但 Reranker 等高精度模型又很贵——这就出现了「召回太粗、精度不够」与「精排太贵、跑不动」之间的矛盾。两阶段索引（Two-Stage Indexing）就是把检索拆成两步：①第一阶段用廉价的 Embedding 做「粗排召回」，从百万级语料里捞出 Top-K（比如 50100）；②第二阶段用 Reranker 在 Top-K 上做「精排」，把分数最高的 Top-N（比如 310）交给 LLM。

    优点缺点#

    优点：①兼顾召回广度与精度，是工业级 RAG 的事实标准；②Reranker 跑的样本量小，延迟与成本都压得住；③可按业务灵活调 K 与 N，平衡效果与开销。缺点：①两个阶段的超参数（Embedding 模型、Top-K、Reranker 选择、Top-N）相互耦合，调优比单阶段复杂；②如果第一阶段 Embedding 召回率本身很差，再怎么精排也救不回来，需要监控两段指标；③对实时性要求高的场景，两阶段延迟叠加可能不可接受。

    术语别名#

    • Two-Stage Indexing
    • 两阶段检索
    • 召回 + 精排
    • Retrieve-and-Rerank

    向量数据库#

    Vector Database（向量数据库）是专门存储高维向量并支持「按相似度检索」的数据库。和传统关系数据库不同，它的查询是「返回与查询向量最相近的 Top-K 条记录」。Milvus、pgvector、Weaviate、Qdrant、Chroma 是常见实现。

    出现动机#

    传统数据库擅长按字段精确匹配，但 RAG 的核心需求是「找出与这段话意思最相近的几条记录」——这是高维向量空间里的相似度问题，传统 SQL 与全文索引都做不好。向量数据库就是为了把「按语义相似度检索」这件事工程化。

    优点缺点#

    优点：①亿级向量下毫秒级 Top-K 召回；②支持 IVF / HNSW / DiskANN 等多种 ANN 索引；③多数支持标量字段过滤。缺点：①相比关系数据库，索引构建成本与内存占用更高；②不同向量库查询语法差异较大；③相似度阈值需要业务反复调优。
